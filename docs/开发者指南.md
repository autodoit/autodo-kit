## 1. 序言：愿景与核心架构 (The AOK Blueprint)

### 1.1 愿景与定位
`autodo-kit` (以下简称 AOK) 的定位是 **“自动化任务领域的标准弹药库”**。它不负责复杂的流转调度（那是 `autodo-engine` 的职责），也不负责状态持久化（那是 `task-db` 的职责）。AOK 的唯一目标是提供经过严格工程化、具有高鲁棒性的 **原子工具 (Tools)** 和 **标准事务 (Affairs)**。

在 v0.5.0 版本中，我们彻底告别了基于字符串反射的 `invoke_capability` 模式，全面推行 **“函数直接调用”** 模式。这一转变旨在提升 IDE 的开发体验（类型补全、重构支持）以及 AI Agent 的调用准确度（减少幻觉、节省 Token）。

### 1.2 核心架构模型：三层工具链
AOK 的 `autodokit/tools/` 目录按照逻辑复杂度和调用深度分为三层，形成了一个清晰的依赖漏斗：

| 层级 | 目录 | 职责 | 设计原则 |
| :--- | :--- | :--- | :--- |
| **原子层 (Atomic)** | `autodokit/tools/atomic/` | 基础功能实现（PDF、文献、LLM 等） | **职责单一、无侧开销**、0 业务感知 |
| **适配层 (Adapters)** | `autodokit/tools/adapters/` | CLI 包装、Agent Payload 处理、Runner 包装 | **环境透明、协议转换**、路径预处理 |
| **分组导出层 (Grouped)** | `autodokit/tools/__init__.py` | 统一公开/开发 API 清单 | **语义聚合、隔离出口**、1 秒定位 |

### 1.3 阿里百炼大模型路由：五层目标架构

为彻底解决“各事务/各工具各写各的模型选择逻辑”问题，AOK 对阿里百炼调用统一采用五层路由架构。

#### 第一层：模型目录层（Catalog）

职责：维护模型元数据单一事实源。

- 模型能力：文本/视觉/代码/推理等。
- 规格信息：上下文上限、思考模式支持、地域限制。
- 成本信息：阶梯价格、可用折扣分类。
- 生命周期：稳定版/最新版/快照版、下线替代映射。

要求：

- 新模型接入优先更新 Catalog，不在事务脚本硬编码。
- 下线模型通过替代映射自动迁移。

#### 第二层：策略层（Policy）

职责：把业务需求翻译成路由决策。

- 输入：`task_type`、`quality_tier`、`budget_tier`、`latency_tier`、`risk_level`、`region`。
- 输出：`primary_model + fallback_chain` 与原因码。

要求：

- 业务层只声明策略意图，不直接决定具体模型。

#### 第三层：估算层（Estimator）

职责：根据输入规模估算成本区间，辅助策略选择。

- 输入长度估算（字符/Token 档位）。
- 阶梯计费区间估算。
- 成本优先/质量优先/时延优先权重计算。

要求：

- 估算结果必须可被审计输出，用于后续策略调优。

#### 第四层：调用门面层（Router Facade）

职责：对外提供稳定 API，屏蔽内部路由细节。

- 推荐入口：`resolve_model_plan(...)`、`invoke_aliyun_llm(...)`。
- 兼容入口：`load_aliyun_llm_config(...)`（内部仍走路由层）。

要求：

- affairs/tools 禁止绕过 Facade 直接写模型选择分支。

#### 第五层：观测治理层（Observability + Governance）

职责：保障路由可观测、可审计、可演进。

- 路由决策日志（命中、fallback、失败原因）。
- 成本估算与实际调用统计。
- 模型更新后的影响面追踪。

要求：

- 每次调用至少记录一次可追溯路由摘要。

### 1.4 路由开发约束（强制）

1. 事务脚本中禁止出现 `if budget: model=qwen...` 这类业务内路由分支。
2. 默认情况下不要在 `affair.json` 中写死 `model`，应使用策略字段。
3. 必须支持“显式锁模型”豁免机制（用于实验复现），并输出审计记录。
4. 涉及 LLM 调用能力变更时，必须同步更新：
  - `docs/API手册.md`
  - `docs/用户手册.md`
  - `docs/开发者指南.md`

### 1.5 路由迁移实施清单（P3-P6）

当你把存量事务/工具从固定模型迁移到统一路由时，按以下顺序执行：

1. 把默认 `model` 从具体型号改为 `auto`。
2. 将事务配置迁移为 `model_route` 策略字段，不在业务代码内写模型分支。
3. 对工具侧统一使用：
  - `resolve_model_plan(...)`
  - `invoke_aliyun_llm(...)`
  - 或 `load_aliyun_llm_config(model="auto", route_hints=...)`
4. 增加静态治理扫描并纳入回归：

```powershell
python scripts/check_aliyun_routing_compliance.py
```

5. 对关键链路运行最小回归（至少覆盖文本、长文本、视觉三类任务）。

### 1.6 静态治理口径

`scripts/check_aliyun_routing_compliance.py` 当前阻断以下模式：

- 代码内把 `model` 默认值写成 `qwen-*`。
- 代码内把 `or "qwen-*"` 作为兜底模型。
- `affair.json` 里把 `"model"` 默认写成具体型号。

推荐把该脚本作为 CI 或提交前本地检查步骤，避免新事务回退到“各写各的模型选择”。

### 2.0.1 在线检索模块收口约定

在线检索相关实现已统一放入 `autodokit/tools/online_retrieval_literatures/`，并由 `online_retrieval_router.py` 作为唯一调度入口。

开发约束：

1. 不要在该目录外新增在线检索程序或配置文件。
2. 不要在业务侧直接 import 或调用 `zh_cnki_*`、`en_open_access_*` 这类子模块作为公开入口。

4. 新增规则优先写入 `autodokit/tools/online_retrieval_literatures/config.json`，再由 router 注入到运行时配置。
5. 若新增单条/批量执行器，必须保证它们仍可被 router 接管，不能新增平行入口。

补充分层边界：

- `online_retrieval_literatures/` 只放与在线检索强耦合、后续难以被其他文献功能复用的专用实现。

- 如果未来文献导入、题录清洗、候选池构建等模块也需要批处理 Bib，就应复用这个通用工具，而不是在 `online_retrieval_literatures/` 再复制一份。

分层约束（2026-04 起）：

1. 路由层（`online_retrieval_router.py`）：只负责入口治理、规则注入和 debug 编排，不承载具体业务分支。
2. 解析层（`online_retrieval_resolver.py`）：负责输入归一化，统一处理 `entries/records/seed_items/cite_keys/pdf_paths`。
3. 功能层（`online_retrieval_service.py` + 各执行器）：负责 source/mode/action 分发和具体检索下载抽取。

新增扩展要求：

- 新增输入形态时，优先扩展解析层，不要把兼容逻辑堆到 router。
- 新增执行器时，优先在功能层增加分发，再复用现有执行器；禁止直接在 router 里引入大段 if/else 业务逻辑。
- 功能层允许根据解析层结果自动补齐单篇/批量执行所需字段，但不得改变已有 payload 关键字段语义。
- 涉及联网能力的回归在目标网络设备执行；本地离线开发阶段至少完成静态导入检查与字段契约检查。

### 2.1 原子层 (`atomic/`)：模块化标准
这是最底层的实现。这里的代码只负责 **“把一件具体的事做到极致”**。
- **无侧开销原则**：导入该层模块不应触发重型初始化（如启动浏览器、加载大型模型）。开销必须延迟到具体的执行函数中。
- **路径处理铁律**：原子层函数**严禁包含**任何路径转换逻辑。它们必须接收 **绝对路径** (`pathlib.Path`) 类型的参数。

#### 代码示例：标准 PDF 原子工具
```python
"""PDF 原子信息提取模块。"""

from typing import Dict, Any

def extract_pdf_info_atomic(pdf_path: Path) -> Dict[str, Any]:
    """提取 PDF 基础元数据（无第三方大型依赖）。

    Args:
        pdf_path: PDF 文件的绝对路径。

    Returns:
        Dict[str, Any]: 包含标题、页数等信息的字典。
    """
    if not pdf_path.is_absolute():
        raise ValueError(f"原子工具必须接收绝对路径，当前为: {pdf_path}")
    
    # 业务细节实现...
    return {"title": "Sample", "pages": 12}
```

### 2.2 适配层 (`adapters/`)：环境与协议中转
适配层负责抹平不同运行环境的差异。
- **CLI 适配**：将普通函数参数映射为命令行选项。
- **Agent 适配**：解析大模型生成的半结构化 JSON Payload，并调用 `autodo-engine` 的路径工具将其中的相对路径转为绝对路径。
- **路径预处理全链路**：这是适配层的重中之重。它负责确保 `atomic` 层见到的永远是无歧义的执行环境。

### 2.3 分组导出层 (`__init__.py`)：统一契约管理


### 2.7 MonkeyOCR Windows GPU 解析工具

本仓库已把 Windows 原生 MonkeyOCR 单篇解析封装到 `autodokit/tools/monkeyocr_windows_tools.py`，并通过 `autodokit.tools` 统一导出。

设计目标：

1. 把“安装依赖、下载权重、写配置、启动解析、收集产物”收敛成一个可复用工具层能力。
2. 避免在沙盒脚本里重复维护同一套 Windows/GPU 启动逻辑。
3. 维持与仓库既有工具风格一致：路径在工具外部预处理为绝对路径，工具内部只做执行编排。

实现位置：

- 工具实现：`autodokit/tools/monkeyocr_windows_tools.py`
- 统一导出：`autodokit/tools/__init__.py`
- 兼容薄封装：`sandbox/test monkey ocr/run_monkeyocr_batch.py`

关键行为：

1. `prepare_monkeyocr_windows_runtime(...)` 会安装 `huggingface_hub`、`triton-windows<3.4>`，按需安装 `modelscope`，然后下载 MonkeyOCR 模型。
2. `run_monkeyocr_windows_single_pdf(...)` 会生成本地 `model_configs.local.yaml`，设置 `MONKEYOCR_DEVICE=cuda`、`CUDA_VISIBLE_DEVICES=0` 等环境变量，并调用 MonkeyOCR 的 `parse.py`。
3. 实时解析日志会写入 `parse_direct_run.log`，同时按页输出图片数打印 `[PROGRESS]`，便于判断任务是否卡住。
4. 解析产物会按单篇目录归档，输出文件名保持固定契约，方便文档、测试和后续事务复用。

Windows + Python 3.13 注意事项：

- 官方 MonkeyOCR Windows 指南要求 `triton-windows<3.4`。
- 我们在验证过程中遇到过 `flash_attn` 在 Windows 上无可用轮子的问题，因此当前仓库的 MonkeyOCR 模型加载路径支持在 CUDA 下自动回退到 SDPA。
- 如果你在新设备/新工作区复现，优先复用本仓库当前的 MonkeyOCR 兼容补丁；若回到纯上游仓库且没有相同补丁，则可能需要改用可安装 `flash_attn` 的环境或继续保留 SDPA 回退策略。

输出契约解释：

- `*.md`：最终解析文本，面向人工阅读或后续笔记化。
- `*_content_list.json`：元素清单，适合程序化处理。
- `*_middle.json`：中间结构，适合质量检查与排错。
- `*_model.pdf`、`*_layout.pdf`、`*_spans.pdf`：模型、布局、span 的可视化核查产物。
- `images/`：页面图片缓存。
- `model_configs.local.yaml`：当次运行配置快照。
- `parse_direct_run.log`：实时执行日志。

扩展建议：

1. 如果要支持新的输出结构，优先改 `autodokit/tools/monkeyocr_windows_tools.py` 的返回契约，再同步更新 API 手册。
2. 如果要支持新的运行平台，优先增加参数，不要把平台判断散落到沙盒脚本里。
3. 若要把 MonkeyOCR 解析纳入正式事务链，请在事务配置中保存 `input_pdf`、`output_dir`、`monkeyocr_root`、`models_dir` 和 `config_path` 五个关键路径。

自 SQLite 主库迁移完成后，AOK 对文献库与知识库采用以下主契约：

### 2.7 PDF 结构化与分块主契约

当前 PDF 主链的实现约束如下：

- 单篇结构化结果统一使用 `aok.pdf_structured.v3`。
- 结构化状态必须写回 `literatures` 的 `structured_*` 字段，而不是只落盘文件不登记主库。
- chunk 主链必须以 `chunk_manifest.json + chunk_shards/*.jsonl + literature_chunk_sets/literature_chunks` 为准。
- 旧 `docs.jsonl`、单一 `chunks.jsonl` 只能作为兼容输入或导出，不再作为新功能默认契约。

开发要求：

- 当前仓库只保留两套 PDF 能力：传统结构化套件、阿里百炼多模态套件。
- 传统结构化套件负责 `aok.pdf_structured.v3`、chunk manifest 与 SQLite 状态回写，后端可为 `local_pipeline_v2` 或 `babeldoc`。
- 阿里百炼多模态套件负责页图理解、结构树、线性索引、重建 Markdown 与多模态 chunks，公开入口固定为 `aok_pdf_aliyun_multimodal_parse.py` 与 `aok_pdf_aliyun_multimodal_batch_manage.py`。
- 不再保留第三套历史实验 PDF 入口；新增 PDF 能力只能并入上述两套之一。
- A05、A06、单篇粗读、单篇精读、综述草稿生成等消费者，应优先从 structured 输入或 `content.db` 登记的 structured 资产读取，缺失时才回退导出物，不再依赖数据库 current view。
- 若扩展结构化 schema、`structured_*` 字段或 chunk 索引表，必须同步更新 `docs/API手册.md`。

两套套件的开发边界：

- 传统结构化套件：面向事务链和主库治理，强调可批量、可回写、可与现有 affairs 协同。
- 阿里百炼多模态套件：面向高精度单篇解析与批量管理，强调单篇独立会话、独立输出目录、结构树与索引产物完整。
- 两套套件共享底层通用工具时，允许复用 `llm_clients.py`、`pdf_page_image_tools.py`、`pdf_elements_extractors.py`、`pdf_multimodal_tree_builder.py` 等基础能力，但对外入口必须分离。

## 3. 事务 (Affair) 生态全解析

事务是 AOK 的对外交付单元，由 **“三件套”** 组成：`affair.json` (定义)、`affair.py` (实现)、`affair.md` (说明)。

### 3.1 三位一体的耦合逻辑
- **`affair.json` (接口契约)**：定义输入/输出 Schema。严禁定义调度控制字段（如重试、超时），这些由引擎处理。
- **`affair.py` (业务心跳)**：核心是 `execute(config_path: Path)` 函数。它调用 `tools` 层的能力，管理事务内部的业务流程。事务应尽量保持 **无状态 (Stateless)**。
- **`affair.md` (语义补全)**：这是面向 AI Agent 的核心资源。它用自然语言解释了该事务的业务边界、前置条件和最佳实践，是 AI 进行意图匹配的关键参考。

## 4. 路径治理与环境合规 (Path Governance)

在多根/分布式工作区中，路径是最大的隐患。AOK 实行 **“绝对路径铁律”**。

### 4.1 核心获取链路
1. **引擎提供基准**：`autodo-engine` 在启动时通过工作区协议确定 `workspace_root`  
2. **调度预转换**：在事务执行前，调度器预先将配置中所有的相对路径，基于 `workspace_root` 转换为绝对路径。  
3. **事务纯消费**：`affair.py` 与 `atomic/` 模块只消费转换后的绝对路径，不负责计算路径。

### 4.2 为什么禁止在代码内写死路径？
任何硬编码路径（即使是 `../../`）都会导致：
- 云端/容器化部署失败。
- 换一台机器或更改一个父目录就彻底瘫痪。
- `deploy.py` 无法正确投影内容。

## 5. 开发协同与工程规范

### 5.1 废弃代码处理：不留痕体原则
项目推行 **“非向后兼容”** 的开发风格：
- 修改或弃用内容时，**直接删除**，不保留任何过时代码（如 `old_v1.py`）或补丁文件。
- 这种风格旨在保持代码库的绝对纯净，降低维护负担和 AI 理解成本。

### 5.2 开发者 Demos 编写标准
每个新增的工具必须在 `demos/scripts/` 下提供单文件 Demo。
- **独立可运行**：Demo 必须能脱离引擎独立执行。
- **最小断言**：必须包含基本的输入检查和输出验证逻辑。

### 5.3 文档同步流程
- 更新工具导出清单后，必须同步更新 `docs/API手册.md`。
- 修改事务后，运行 `python scripts/generate_affair_manual.py` 刷新总手册。
- 若扩展了 `autodokit.tools.atomic.task_aok` 的表结构或公开函数，必须同时更新 API 手册中的字段约定与扩展表说明，避免文档继续停留在旧版双表结构。
- 若切换或新增文献库/知识库主库接口，必须同步更新 `docs/用户手册.md`、`docs/开发者指南.md` 与相关迁移指南，明确 SQLite 为主库、CSV 为导出/迁移格式。

### 新增模块：本地 Git 快照与极简任务账本（开发者备注）

文件位置：`autodokit/tools/atomic/task_aok/git_snapshot_ledger.py`

快速说明：本模块实现本地-only 的 Git 快照与极简 SQLite 账本，用于在事务闸门（pass_next）处做可追溯提交记录与回滚计划登记。为便于协作，开发者需要注意下列要点：

- 单元测试：已增加 `tests/test_git_snapshot_ledger.py`，可通过以下命令单独运行：

```powershell
uv run pytest tests/test_git_snapshot_ledger.py -q
```

- 测试/运行注意：测试会在工作区初始化本地 Git（`git init`）、`git add` 与 `git commit`，因此请在隔离的临时工作区或 CI runner 中运行，避免把个人未提交变更误 commit。模块内已设置默认 `GIT_AUTHOR_*`/`GIT_COMMITTER_*` 环境变量以降低因缺少全局 git config 导致的失败概率。

- 集成建议：把 `git_create_snapshot_for_task` 在事务的 `pass_next` 回调中以非阻塞方式调用；示例：

```python
from autodokit.tools.atomic.task_aok.git_snapshot_ledger import git_create_snapshot_for_task

def on_pass_next(workspace_root, task):
  try:
    git_create_snapshot_for_task(
      workspace_root,
      task_uid=task["task_uid"],
      workflow_uid=task.get("workflow_uid", ""),
      node_code=task.get("node_code", ""),
      gate_code=task.get("gate_code", "pass_next"),
      includes_attachments=True,
    )
  except Exception as e:
    # 记录但不阻断主链
    from autodokit.tools.atomic.log_aok import append_aok_log_event
    append_aok_log_event(project_root=workspace_root, event_type="snapshot.failed", payload={"error": str(e)})
```

- 回滚策略说明：当前 `git_rollback_by_task_uid` 仅做目标快照的查询与回滚记录登记（`preview` / `planned` / `done` 状态）。若需要自动执行恢复，建议实现受控流程：创建临时分支(`git checkout -b restore-{task_uid} {commit_hash}`)、人工或自动化校验、再决定是否合并或强制 reset。任何自动执行的破坏性回滚都必须伴随人工确认与审计记录。

- 文档同步：修改或扩展这个模块的接口时，请同时更新 `docs/API手册.md` 与 `docs/用户手册.md`。可使用仓库内的 `update-sync` skill 或手动按 `docs/` 的约定补写内容并提交 PR。
