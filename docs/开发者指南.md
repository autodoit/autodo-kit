# 开发者指南

autodo-kit 的职责是维护官方事务内容库，而不是承载流程引擎。新增或修改事务时，应遵守“事务实现最小化、路径预处理集中化、运行时能力外置化”的边界。

## 1. 目录约束

- autodokit/affairs：每个事务一个目录，包含 affair.py、affair.json、affair.md
- autodokit/tools：仅放事务侧可复用工具
- scripts/generate_affair_manual.py：根据事务元数据生成总手册
- scripts/aob_tools/：承接 AOB 迁移而来的执行型脚本入口

## 2. 事务开发规则

1. 尽量只改事务业务逻辑，不把调度、状态机、任务数据库逻辑写回本仓库。
2. `affair.json` 仅保留纯业务参数，不再写入 runner/domain/node 等管理字段。
3. 路径字段由运行前预处理统一转绝对路径，不在 affair.py 中重复发明路径转换规则。
4. 通用能力优先沉淀到 autodokit.tools，避免在多个事务里复制实现。
5. 事务治理元数据（runner、node_template、ownership/governance）统一由事务管理系统数据库维护。

## 3. 对引擎的依赖边界

- load_json_or_py、resolve_paths_to_absolute、事务注册与配置加载等基础能力来自 autodo-engine。
- 本仓库通过 autodokit.tools 统一再导出这些能力，事务脚本不直接感知引擎内部目录结构。
- 如果需要新增路径字段，先对齐 autodo-engine 的预处理逻辑，再更新事务实现。

## 4. 文档同步

- affair.json、affair.md、affair.py 变更后，应重新生成 docs/AOK预置事务手册.md。
- README、用户手册、API 手册只说明事务库能力，不再描述 CLI、taskdb 或调度主循环。
- 若变更 `scripts/aob_tools/` 或 `autodokit/tools/gateway/public_tools_manifest.json`，应同步 README、用户手册、API 手册中的主入口与分级说明。

## 5. AOB 迁移脚本与公共工具网关

- `scripts/aob_tools/` 保存从 AOB 收敛到 AOK 的执行型入口。
- `autodokit/tools/gateway/public_tools_manifest.json` 是当前 AOK 公共工具分级清单初版。
- `autodokit.tools.读取公共工具清单(...)`、`列出公共工具(...)`、`获取公共工具(...)` 提供统一只读查询接口。

当前约定：

1. 默认暴露等级为 `internal`。
2. 仅白名单条目进入 `public-read` 或 `public-safe`。
3. 新增工具若准备对外暴露，必须先登记 manifest，再更新相应文档。

## 6. 全量合规迁移工具

- 迁移脚本：`scripts/migrate_affairs_to_compliance.py`
- 作用：将 legacy 事务清单迁移为纯业务 `affair.json`，并生成 `autodo-engine/config/affair_metadata_overrides.json`。
