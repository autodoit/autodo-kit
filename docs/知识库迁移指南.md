# 知识库迁移指南

本文档用于将旧版知识库索引（`notes.csv` / `relations.csv` / `attachments.csv`）迁移到 AOK 004 契约。

## 1. 目标结构

迁移完成后使用以下结构：

- `database/knowledge_index.csv`
- `database/knowledge_attachments.csv`
- `knowledge/`（Obsidian Markdown 笔记主源）

## 2. 迁移命令

先执行预演：

```bash
python scripts/migrate_knowledge_db_to_004.py --project-root . --dry-run
```

确认无误后执行正式迁移：

```bash
python scripts/migrate_knowledge_db_to_004.py --project-root .
```

如需禁用自动备份：

```bash
python scripts/migrate_knowledge_db_to_004.py --project-root . --no-backup
```

## 3. 迁移规则

- 主键统一为 `uid_knowledge`。
- 旧字段 `item_uid` 会映射为 `uid_knowledge`。
- 旧字段 `source_uid` 会映射为 `uid_literature`。
- 若缺少 `uid_knowledge`，会根据 `note_path`/`title` 生成稳定 UID。
- 可选地为缺失 frontmatter 的 Markdown 笔记补最小合法 frontmatter。

## 4. 执行后检查

建议至少检查以下项目：

1. `database/knowledge_index.csv` 行数与迁移日志一致。
2. `database/knowledge_attachments.csv` 中 `uid_knowledge` 都能在索引表找到对应记录。
3. `knowledge/` 目录下关键笔记均含 `uid_knowledge`、`title`、`note_type`、`status`。
4. 运行测试：

```bash
pytest tests/test_knowledgedb.py tests/test_bibliodb.py -q
```

## 5. 回滚建议

- 默认会在迁移前生成 `database/backup_knowledge_legacy/` 备份目录。
- 若迁移结果不符合预期，可删除新表并从备份恢复旧文件，再修正参数后重试。

## 6. 迁移到 SQLite（当前默认主路径）

为提高索引查询、绑定关系一致性与运行时可用性，建议把迁移后的 CSV 导入统一内容主库 `database/content/content.db`。当前迁移脚本会把 `knowledge_index.csv` 与 `knowledge_attachments.csv` 导入该主库中的知识域表，并保留旧 `knowledge.db` 路径作为显式兼容输入。

迁移命令示例：

```bash
python scripts/migrate_knowledge_to_sqlite.py --project-root .
```

只预演：

```bash
python scripts/migrate_knowledge_to_sqlite.py --project-root . --dry-run
```

迁移后仍保留 CSV 作为可导出的分析格式；但知识管理系统的默认运行时主库已切换为统一内容主库 SQLite。
