# 文献库迁移指南

## 1. 目标

将旧文献库结构：

- `database/references/literature_items.csv`
- `database/references/literature_files.csv`

迁移为 AOK 003 新结构：

- `database/references/literatures.csv`
- `database/references/literature_attachments.csv`

对应迁移脚本：`scripts/migrate_literature_schema_003.py`。

## 2. 迁移前检查

- 确认项目根目录存在 `database/references/`。
- 确认旧 CSV 编码为 UTF-8/UTF-8-SIG。
- 建议先执行 dry-run 查看行数变化。

## 3. 使用方式

在仓库根目录执行：

- 仅预演，不写入：
  - `python scripts/migrate_literature_schema_003.py --project-root . --dry-run`
- 正式迁移（默认会备份旧 CSV）：
  - `python scripts/migrate_literature_schema_003.py --project-root .`
- 正式迁移且不备份：
  - `python scripts/migrate_literature_schema_003.py --project-root . --no-backup`

## 4. 字段映射摘要

### 4.1 主表映射（旧 -> 新）

- `item_uid` -> `uid_literature`
- `title` -> `title`
- `authors` -> `authors`
- `year` -> `year`
- `source_type` -> `source_type`
- `origin_path` -> `origin_path`
- `created_at` -> `created_at`
- `updated_at` -> `updated_at`

新增派生字段：

- `cite_key`
- `clean_title`
- `title_norm`
- `first_author`
- `entry_type`（默认 `article`）
- `is_placeholder`（默认 `0`）
- `has_fulltext`
- `primary_attachment_name`
- `standard_note_uid`（默认空）

### 4.2 附件表映射（旧 -> 新）

- `file_uid` -> `uid_attachment`
- `item_uid` -> `uid_literature`
- `file_path` -> `attachment_name` / `storage_path` / `source_path`
- `file_type` -> `attachment_type`
- `checksum` -> `checksum`
- `created_at` -> `created_at`

新增派生字段：

- `file_ext`
- `is_primary`（PDF 默认 `1`）
- `status`（默认 `active`）
- `updated_at`

## 5. 风险与回滚

- 风险：迁移后新旧表可能并存，若后续事务仍误读旧路径会造成数据分叉。
- 风险：旧表字段空值较多时，部分派生字段可能为默认值。

回滚建议：

- 若保留了默认备份文件（`*.bak-时间戳`），将备份文件改回原名即可恢复旧结构。
- 若未备份，请在迁移前自行复制 `database/references/` 目录再执行。

## 6. 建议后续动作

- 迁移后统一检查事务配置默认输出是否指向 `literatures.csv`。
- 运行一次文献导入与语义预筛选链路，确认 `uid_literature` 与 `has_fulltext` 已稳定流转。

## 7. 迁移到 SQLite（当前默认主路径）

为提高可靠性、索引与并发查询能力，建议将 CSV 表导入 SQLite 数据库作为运行时主存储，同时保留 CSV 作为分析友好导出。

- 迁移脚本：`scripts/migrate_literature_to_sqlite.py`
- 目标文件（示例）：`database/content/content.db`。当前至少会落 `literatures`，并把附件真相源写入 `attachments` 与 `literature_attachment_links`；`literature_attachments` 仅保留迁移期兼容投影。

示例：先保证已生成 `literatures.csv` 与 `literature_attachments.csv`（见前面步骤），然后运行：

```bash
python scripts/migrate_literature_to_sqlite.py --project-root .
```

如需只预演：

```bash
python scripts/migrate_literature_to_sqlite.py --project-root . --dry-run
```

迁移后可用 pandas 直接读取 SQLite：

```python
import sqlite3
import pandas as pd
conn = sqlite3.connect('database/content/content.db')
df = pd.read_sql_query('SELECT * FROM literatures WHERE year>2018', conn)
```

注意：SQLite 已是文献管理系统的默认运行时主库；CSV 仅适合作为分析导出或历史迁移中间格式。若你显式传入旧 `references.db` 路径，系统仍会按兼容模式接受，但不再作为默认主契约。

补充说明：

- 已匹配主附件进入当前主库后，物理文件名默认会被规范为稳定附件 UID 派生基名。
- 迁移或导入过程中未命中的孤儿附件，不应继续视作正式附件事实，建议隔离到单独复核目录后再决定是否补录。
- 若后续通过 A040 在线补件，新增下载文件也应进入同一套 `attachments` + `literature_attachment_links` 契约，而不是另起一套文件管理规则。
