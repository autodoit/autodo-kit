# 用户手册

autodo-kit 用于提供官方事务与事务工具，并内置最小运行时。未安装 autodo-engine 时也可直接运行事务。

补充说明：

- AOB 迁移过来的执行能力现在统一由 `autodokit.tools` 承接；
- `scripts/aob_tools/` 与 `scripts/run_*.py` 仅作为参数化薄入口，不再承载核心逻辑。

## 1. 安装

```powershell
uv pip install -e .
```

如需完整工作流调度能力，可额外安装：

```powershell
uv pip install -e ../autodo-engine
```

## 2. 查看与更新事务手册

```powershell
python scripts/generate_affair_manual.py
```

生成结果位于 docs/AOK预置事务手册.md。

## 3. 直调事务

### 3.0 统一后处理执行边界（2026-04）

从本轮版本开始，A010-A160 主链的通用后处理采用“编排层统一触发”模式：

- 不再要求每个事务 `affair.py` 在内部各自执行通用后处理。
- 默认由 PA 编排层在节点结束后统一触发后处理技能。
- 若通过 `autodokit.run_affair(...)` 直调事务，运行时仍提供统一后处理兜底。

建议：

- 主链编排优先使用“PA + 统一后处理技能”口径，保证任务与日志审计一致。
- 单事务本地调试可继续直调 `execute`，但若需要完整审计产物，应补跑统一后处理。

推荐方式：直接导入具体事务脚本并调用 `execute`。

```python
from pathlib import Path
from autodokit.affairs.PDF文件转md文件.affair import execute

workspace_root = Path.cwd()
config_path = workspace_root / "demos" / "settings" / "pdf_to_md.json"
outputs = execute(config_path=config_path, workspace_root=workspace_root)
print(outputs)
```

补充：也可通过桥接入口调用（兼容方式）：

```python
from pathlib import Path
import autodokit as aok

outputs = aok.run_affair(
    "PDF文件转md文件",
    config={
        "pdf_dir": "data/input",
        "output_dir": "data/output",
    },
    workspace_root=Path.cwd(),
)
print(outputs)
```

说明：

- 直接导入事务脚本时，调用链最短，便于针对单事务做调试与问题定位。
- run_affair 默认由 autodo-kit 内置运行时执行。
- 若环境中安装了 autodo-engine，可继续与其调度层配合使用。
- 在进入运行时前所有路径都必须是绝对路径。
- 对 A010-A160 这类主链固定节点，建议同时维护一份显式入口注册表，避免仅靠临时扫描再推断 runner。

### 3.1 在线检索文献模块的唯一入口

在线检索文献模块现在统一收敛到 `autodokit/tools/online_retrieval_literatures/` 目录，用户只允许通过 `autodokit.tools.run_online_retrieval_router(...)` 或 `autodokit.tools.online_retrieval_literatures.online_retrieval_router.route(...)` 调用。

说明：

- 在线检索相关的程序、配置文件和实现模块都应放在 `autodokit/tools/online_retrieval_literatures/` 下维护。
- `online_retrieval_router.py` 是唯一对外入口；不要直接调用其中的子模块执行下载、抽取或检索。
- 默认规则配置文件是 `autodokit/tools/online_retrieval_literatures/config.json`。
- 若需要调整跳过规则，应优先改该配置文件，而不是在单个子模块里临时分支。

实现分层（2026-04 起）：

- 请求画像层：`profiles/request_profile.py`
- 路由层：`router/route_entry.py`、`online_retrieval_router.py`
- 编排层：`orchestrators/*.py`、`policies/*.py`、`catalogs/*.py`
- 执行层：`executors/*.py`、`zh_cnki_*`、`en_open_access_*`

用户可见的稳定入口只有两类：

1. `run_online_retrieval_router(...)`：推荐入口。
2. `run_online_retrieval_from_bib(...)`：Bib 批处理入口，内部仍转到 router。

用户不应直接依赖内部文件，例如：

1. `online_retrieval_resolver.py`
2. `online_retrieval_service.py`
3. `school_foreign_database_portal.py`
4. `en_chaoxing_portal_retry.py`

这些文件是开发和排障时使用的内部实现，不是用户级 API。

专题文档：

- 在线检索文献模块专题：`docs/在线检索文献模块专题.md`

### 3.2 Bib 批处理在线检索

如果你手里已经有一个 Bib 文件，并且希望按每条记录批量执行中文下载原文 PDF、英文下载原文 PDF、中文 HTML 结构数据、英文 HTML 结构数据，推荐直接调用通用工具 `run_online_retrieval_from_bib(...)`。

这个工具不放在 `online_retrieval_literatures/` 里，因为它本质上是通用的 Bib 批处理与记录匹配工具，未来也可以被文献导入、题录清洗、候选池构建等功能复用。它只是把 Bib 任务翻译成在线检索路由 payload，然后继续走在线检索三层结构。

示例：

```python
from autodokit.tools import run_online_retrieval_from_bib

result = run_online_retrieval_from_bib(
    {
        "bib_path": r"sandbox/20260407/测试用.bib",
        "output_dir": r"sandbox/20260407/router_bib_tasks",
        "max_pages": 1,
        "en_per_page": 20,
        "max_entries": 0,
        "use_llm_matching": True,
        "llm_api_key_file": r"demos/settings/配置文件/bailian_api_key.txt",
    }
)

print(result["artifacts"])
```

如果你只想保留最薄的命令行入口，也可以继续使用 `sandbox/20260407/run_bib_online_retrieval_tasks.py`，它只是把命令行参数转交给这个工具。

### 3.3 Windows 上用 3060 跑 MonkeyOCR 单篇 PDF

本仓库已经把 Windows 原生解析能力封装成 AOK tool，可直接调用，不需要手工拼 `parse.py` 命令。

成功解析时采用的配置如下：

- 系统：Windows 11
- GPU：NVIDIA GeForce RTX 3060
- PyTorch：2.11.0+cu126
- 设备模式：`cuda`
- 模型：`MonkeyOCR-pro-1.2B`
- 模型权重目录：`sandbox/test monkey ocr/model_weight`
- 运行后端：`transformers`
- 批大小：`2`
- 运行时依赖：`triton-windows<3.4`
- 运行环境变量：`MONKEYOCR_DEVICE=cuda`、`CUDA_VISIBLE_DEVICES=0`、`PYTHONUTF8=1`

推荐的直接调用方式：

```python
from pathlib import Path
from autodokit.tools import run_monkeyocr_windows_single_pdf

result = run_monkeyocr_windows_single_pdf(
    input_pdf=Path(r"D:\workspace\sandbox\test monkey ocr\input\“双支柱”调控与银行系统性风险——基于SRISK指标的实证分析.pdf"),
    output_dir=Path(r"D:\workspace\sandbox\test monkey ocr\output"),
    monkeyocr_root=Path(r"D:\workspace\sandbox\MonkeyOCR-main"),
    models_dir=Path(r"D:\workspace\sandbox\test monkey ocr\model_weight"),
    config_path=Path(r"D:\workspace\sandbox\test monkey ocr\model_configs.local.yaml"),
    log_path=Path(r"D:\workspace\sandbox\test monkey ocr\parse_direct_run.log"),
    device="cuda",
    gpu_visible_devices="0",
    ensure_runtime=False,
)
```

如果你在另一台 Win11 设备或另一个 workspace 里复现，优先检查这几项：

1. 安装 CUDA 版 PyTorch，保证 `torch.cuda.is_available()` 返回 `True`。
2. 安装 `huggingface_hub`、`triton-windows<3.4`，然后下载 MonkeyOCR 模型权重。
3. 将 `model_configs.local.yaml` 的 `device` 改成 `cuda`，并把 `models_dir` 指向你的权重目录。
4. 如果你要手工跑官方仓库，建议继续保留本仓库的 FlashAttention 回退逻辑；否则在 Python 3.13 的 Windows 环境下，官方路径可能因为缺少 `flash_attn` 轮子而失败。

解析完成后，目标输出目录里通常会看到这些文件：

- `*.md`：最终解析结果，适合直接阅读或后续二次处理。
- `*_content_list.json`：页面/元素级内容清单，适合程序消费。
- `*_middle.json`：中间结构化结果，便于排错与再加工。
- `*_model.pdf`：模型阶段可视化图，便于核对版面识别。
- `*_layout.pdf`：布局阶段可视化图，便于检查分区与区域识别。
- `*_spans.pdf`：文本 span 可视化图，便于检查文本块切分。
- `images/`：页面渲染图，供复核与后续人工校验。
- `model_configs.local.yaml`：本次运行使用的本地配置快照。
- `parse_direct_run.log`：实时运行日志，能看到 batch 进度和报错。

如果你只想复现同一套运行环境，直接复用这份目录结构即可：

- `sandbox/MonkeyOCR-main/`：MonkeyOCR 仓库本体。
- `sandbox/test monkey ocr/input/`：待解析 PDF。
- `sandbox/test monkey ocr/model_weight/`：权重目录。
- `sandbox/test monkey ocr/output/`：解析结果目录。

新增可选输入（保持向后兼容）：

- `seed_items`：支持按条目传 `cite_key`、`pdf_path`、`title`、`detail_url`
- `cite_keys`：支持一组 cite_key 批量输入
- `pdf_paths`：支持一组 PDF 绝对路径批量输入
- `content_db`：可选，提供后可辅助补齐标题等检索线索

## 5. 常见问题

1. 直接 import autodokit 后找不到 run_affair：确认安装的是最新 autodo-kit 版本。
2. 事务执行时报路径错误：优先检查 workspace_root 与配置中的路径字段。
3. 手册内容过旧：重新执行 scripts/generate_affair_manual.py。
4. 学校数据库导航清单生成报错：
    - 可先在配置中显式传入 `library_nav_url`（学校图书馆数据库导航网址）重试；
    - 若仍失败，通常是学校站点改版或订阅状态变更，请联系开发者升级适配程序。

## 5.2 文献库与知识库主库说明

当前版本起，AOK 的文献管理系统与知识管理系统默认使用 SQLite 作为主库：

- 默认内容主库：`database/content/content.db`
- 兼容输入路径：`database/references/references.db`、`database/knowledge/knowledge.db`
- `content.db` 的正式契约是“程序只读写表，视图仅供人类读取”。当前库中允许保留中文阅读状态视图等只读视图，用于排查与巡检，但事务与工具不得把视图当作写入目标。

CSV 不再作为运行时主库，仅保留以下用途：

- 历史项目迁移输入
- 审计导出
- pandas 快速分析中间件

常用操作：

```powershell
python scripts/migrate_literature_to_sqlite.py --project-root .
python scripts/migrate_knowledge_to_sqlite.py --project-root .
```

如需从 SQLite 导出 CSV 用于分析，可直接使用 `pandas.read_sql_query(...)` 或调用对应导出工具。

如果你要重新初始化一个全新的工作区，A010 的主入口仍是 `C:\Users\Ethan\.copilot\skills\A010_项目初始化_v5\scripts\generate_config.py`。当前技能内容版本已升级到 v6：脚本会一次性生成工作区配置、事务配置、任务账本、日志目录与 `content.db`，并统一把事务产物目录落到 `workspace/tasks/`。当前口径下，`content.db` 允许保留人类只读视图；初始化与自检应关注“主链程序是否只读写表、视图是否保持只读用途”，而不是强行要求零视图。

补充说明：

- A070 到 A105 的正式当前态真相源统一为 `literature_reading_state`，不再把 `literature_reading_queue` 当作新项目默认契约。
- PDF 结构化目录统一映射到 `workspace/references/structured_monkeyocr_full`。
- 初始化默认只生成 snapshot 计划；如需提交，先由人类确认。

补充说明：

- `literature_tags` 已经是文献-标签的 n 对 n 关系表。

说明：新项目与默认事务配置都会优先使用统一内容主库；仅当你显式传入旧路径参数时，系统才会按兼容模式接受原 `references.db` / `knowledge.db` 路径。

## 5.3 AOK 全局日志配置

如果你使用工作区配置文件运行 AOK 主链，日志开关写在 `workspace/config/config.json` 的顶层 `logging` 字段中。

最小可用示例：

```json
{
    "logging": {
        "enabled": true,
        "snapshot_mode": "log_only"
    },
    "paths": {
        "log_db_path": "D:/your-project/workspace/database/logs/aok_log.db",
        "logs_dir": "D:/your-project/workspace/logs"
    }
}
```

字段说明：

- `logging.enabled`：是否允许 AOK 写运行日志。`true` 表示写入，`false` 表示关闭。
- `logging.snapshot_mode`：日志快照模式。当前模板默认使用 `log_only`，普通用户保持默认即可。
    可选值：

    - `"log_only"`：仅记录结构化 SQLite 日志，不保存运行时快照（默认）。
    - `"log_and_snapshot"`：在记录 SQLite 日志的同时，保存运行时快照到 `runtime_dir`（用于故障复现与审计）。
- `paths.log_db_path`：AOK 结构化日志数据库路径，通常指向 `workspace/database/logs/aok_log.db`。
- `paths.logs_dir`：运行日志与审计文件目录，通常指向 `workspace/logs`。

常见配置方式：

1. 开启日志：

```json
"logging": {
    "enabled": true,
    "snapshot_mode": "log_only"
}
```

2. 关闭日志：

```json
"logging": {
    "enabled": false,
    "snapshot_mode": "log_only"
}
```

使用建议：

- 只想控制开关时，优先改 `logging.enabled`。
- 想改日志写到哪里时，再改 `paths.log_db_path` 和 `paths.logs_dir`。
- 新项目如果直接使用初始化模板，默认配置文件里已经带有 `logging` 字段，可以直接把 `enabled` 从 `false` 改成 `true`。

注意：

- 不要把 `paths.log_db_path` 误当成开关；它只决定日志数据库写到哪里。
- 如果你的某些工具链依赖全局配置读取日志开关，显式写出 `logging.enabled` 会比省略该字段更稳妥。

## 5.4 工作区迁移后的路径重写

如果你把整个 workspace 迁移到新设备或新目录，但旧的绝对路径仍然写在配置、步骤产物或 SQLite 里，建议先执行路径 inventory，再执行改写。

推荐使用的 skill：`.opencode/skills/workspace-path-migration/`

推荐流程：

1. 先运行 `inventory-only`，查看所有命中的旧路径清单。
2. 再确认 `old=>new` 映射是否完整，尤其是 `workspace_root` 和仓库根路径。
3. 最后使用 `--apply` 真正写入。

调用示例：

```powershell
c:/Users/Ethan/CoreFiles/ProjectsFile/autodo-kit/.venv/Scripts/python.exe .opencode/skills/workspace-path-migration/scripts/run_workspace_path_migration.py \
    --workspace-root c:/Users/Ethan/CoreFiles/ProjectsFile/AcademicResearch-auto-workflow/workspace \
    --mapping "C:/Users/Ethan/CoreFiles/ProjectsFile/AcademicResearch-auto-workflow/workspace=>D:/Research/workspace" \
    --mapping "C:/Users/Ethan/CoreFiles/ProjectsFile/AcademicResearch-auto-workflow=>D:/Research" \
    --inventory-only \
    --report-path c:/Users/Ethan/CoreFiles/ProjectsFile/AcademicResearch-auto-workflow/workspace/logs/path_migration_inventory.json
```

说明：

- 默认会排除 `.copilot`、`C:/Windows`、`C:/Program Files` 和外部盘路径前缀，减少误改风险。
- `inventory-only` 只做扫描，不会写回文件或 SQLite。
- `--apply` 需要在确认清单后再使用。

你可以先把命中的路径理解为三类，再决定怎么处理：

| 分类 | 处理建议 | 典型内容 |
| --- | --- | --- |
| 可迁移 | 默认改写 | `config.json` 的 `workspace_root`、`paths.*`；各 `affairs_config/*.json` 的 `output_dir`、`bibtex_path`、`pdf_dir`、`sqlite_db_path`；SQLite 里的 `pdf_path`、`origin_path`、`structured_abs_path`、`storage_path`、`source_path`、`note_path`、`artifact_path`、`gate_review_path`、`manifest_path`、`chunks_abs_path`、`shard_abs_path` |
| 只读引用 | 可选改写 | `steps/`、`runtime/`、`knowledge/`、`references/structured_*` 下的 JSON/CSV/Markdown 产物里记录的 `note_path`、`pdf_path`、`structured_path`、`normalized_structured_path`、`output_dir`、`config_path`、`workspace_root` |
| 外部资源 | 默认跳过 | `origin_attachments_root`、`template_root`、`venv_path`、用户级 `.copilot`、系统盘前缀、工作区外的数据目录 |

边界上最重要的一点是：`content.db` 不是唯一的路径来源。`steps/`、`runtime/`、`knowledge/` 和 `references/` 里生成的中间结果也经常把绝对路径写回去，所以 inventory 时要一起看。

### 本地 Git 快照与极简任务账本（用户向简要）

概述：AOK 新增了“本地 Git 快照 + 极简任务账本”能力，便于在事务闸门通过时生成可追溯的代码/产物快照并登记回滚计划。该能力为本地-only，不会自动 push 到任何远端仓库。

快速上手（最小示例）：

```python
from pathlib import Path
from autodokit.tools.atomic.task_aok.git_snapshot_ledger import ledger_init, git_create_snapshot_for_task, git_rollback_by_task_uid

root = Path.cwd()
ledger_init(root)
res = git_create_snapshot_for_task(
    root,
    task_uid="task-20260403-001",
    workflow_uid="wf-xyz",
    node_code="nodeA",
    gate_code="pass_next",
)
print('快照已写入：', res['summary_path'])

# 预览回滚
preview = git_rollback_by_task_uid(root, source_task_uid="task-20260403-002", target_task_uid="task-20260403-001", mode="preview")
print(preview)
```

产物位置说明：

- 账本（SQLite）：`workspace_root/database/tasks/tasks.db`
- 快照摘要（JSON）：`workspace_root/logs/git_snapshots/{task_uid}.json`
- 本地 tag：`aok/task/{task_uid}`（仅本地）

提醒：

- `git_rollback_by_task_uid` 默认仅做“预览/登记”，不会自动回滚工作区；若需执行回滚，请按审计与人工确认流程执行 `git checkout` / `git reset` 等命令。
- 快照默认包含工作区当前文件状态；`.gitignore` 只忽略 `database/logs/aok_log.db`，其余内容包括附件默认纳入 Git。

非编程用户建议：使用 demo 脚本或由开发者在事务 pass_next 钩子中开启自动快照记录，避免手工操作产生误提交。

## 7. 阿里百炼大模型统一路由（用户向）

从本轮改造开始，AOK 中涉及阿里百炼大模型调用的事务与工具会统一走路由层，不再建议在业务配置里手写固定模型名。

### 7.1 你需要关心的配置

普通使用场景建议只关心“策略意图”，不直接绑定具体型号：

- `routing_profile`：路由场景预设（例如：`balanced_default`、`cost_first`、`quality_first`）。
- `quality_tier`：质量档位（建议：`standard`、`high`）。
- `budget_tier`：成本档位（建议：`cheap`、`balanced`、`premium`）。
- `latency_tier`：时延档位（建议：`low`、`medium`、`high`）。
- `region`：推理地域（例如：`cn-beijing`）。

说明：

- 若不填具体 `model`，系统会按事务类型、输入规模和预算档位自动选模型。
- 若你填写 `model`，系统会视为“显式锁模型”，仅在你明确需要复现实验时建议使用。

### 7.2 最小可用示例（推荐）

```json
{
    "routing_profile": "balanced_default",
    "quality_tier": "standard",
    "budget_tier": "balanced",
    "latency_tier": "medium",
    "region": "cn-beijing"
}
```

### 7.3 什么时候不该手写 model

以下场景优先不要手写 `model`：

- 常规生产任务（避免官方模型更新后你仍停留在旧型号）。
- 成本敏感任务（系统会根据输入规模与预算自动调度更合适模型）。
- 需要跨事务统一策略的任务链（防止每个事务各写各的导致行为不一致）。

### 7.4 什么时候可以手写 model

只有以下场景建议手写：

- 可复现实验（必须固定快照版本）。
- 灰度验证（对比新旧模型效果）。

此时建议同时记录：

- 为什么要锁模型；
- 预计解锁时间；
- 回滚到自动路由的条件。

### 7.5 从固定 model 迁移到策略路由

推荐按下面顺序迁移：

1. 把事务配置中的 `model` 改为 `"auto"`。
2. 新增或补齐 `model_route` 字段，至少声明：
    - `task_type`
    - `quality_tier`
    - `budget_tier`
    - `latency_tier`
3. 在回归阶段执行路由合规扫描：

```powershell
python scripts/check_aliyun_routing_compliance.py
```

执行结果说明：

- `finding_count = 0`：表示未发现新的硬编码模型默认值。
- `finding_count > 0`：请根据输出中的 `file_path + line` 定位并改为 `auto + model_route`。

### 7.6 日常运维建议

- 优先通过 `budget_tier` 控制成本，不要在事务里手写低价模型名。
- 若线上表现波动，先提升 `quality_tier` 或调整 `latency_tier`，再考虑短期锁模型。
- 视觉类或长文本任务仍建议显式提供 `task_type`，便于路由层正确挑选候选模型。

也可以直接调用工具 API：

```python

code = run_aob_workflow_deploy(
    workflow_id="academic",
    engine_ids=["opencode"],
    target_dir=r"D:\\ProjectS",
    dry_run=True,
)
print(code)
```

## 附录：AOK CLI 工具（aok_tool）简介

为了便于在 CI / 脚本或轻量化环境中使用，本项目新增一个轻量 CLI 工具 `aok`，位于 `tools/aok_tool/aok.py`。
该工具目前提供两个子命令：

- `write-log`：把任意文本写入工作区的 AOK 日志数据库（默认 `workspace/database/logs/aok_log.db`）。
- `list-queue`：历史兼容排查命令。用于查看旧项目里 `literature_reading_queue` 指定阶段的条目；新项目应优先查看 `literature_reading_state` 及其中文视图。

安装/调用方式：在仓库根目录使用虚拟环境中的 Python 运行脚本。例如：

```bash
# 列出 A060 队列（最多 50 条）
python tools/aok_tool/aok.py --workspace workspace list-queue --stage A060 --limit 50

# 把本地文件内容写入日志
python tools/aok_tool/aok.py --workspace workspace write-log --message-file ./my_message.txt
```

工具设计说明：

- `--workspace`（或 `-w`）用于指定工作区根目录，便于在不同实例/路径下复用。若不指定则使用当前工作目录（`.`）。
- `write-log` 支持直接从 `--message` 指定文本，或从 `--message-file` 读取 UTF-8 文件，或从标准输入读取。写入时会按 UTC 时间戳写入 `aok_logs` 表。表结构：`id, ts, source, content`。
- `list-queue` 会读取 `<workspace>/database/content/content.db`，并以 JSON 输出 `count` 与 `items` 列表，便于被其它脚本或管道消费。

迁移建议：

1. 旧脚本 `workspace/scripts/write_aok_log.py` 与 `workspace/scripts/list_a060_queue.py` 已迁移到 `tools/aok_tool/`，建议引用新路径或统一在 CI 中调用该 CLI。 
2. 若你在自动化脚本中需频繁写日志或查询队列，优先使用 `aok` CLI，能保证路径解析与日志表结构一致。

示例集成（shell）：

```bash
# 在 CI 中把会话摘要写入日志
python tools/aok_tool/aok.py --workspace workspace write-log --message-file ./ci/session-summary.txt --source ci:session

# 取出 A060 队列并传给后续处理脚本
python tools/aok_tool/aok.py --workspace workspace list-queue --stage A060 --limit 100 > /tmp/a060.json
python scripts/process_a060_batch.py /tmp/a060.json
```

## 11. PDF 结构化主链

当前版本起，PDF 相关主链推荐改为：

1. `PDF文件转结构化数据文件`
2. `解析与分块`
3. `向量化与索引构建`
4. `单篇粗读` / `单篇精读` / `综述草稿生成`

主契约说明：

- 单篇 PDF 的主缓存为 `*.structured.json`，schema 固定为 `aok.pdf_structured.v3`。
- 文献主库会把结构化状态写入 `database/content/content.db` 的 `structured_*` 字段。
- 分块结果主契约为 `chunk_manifest.json + chunk_shards/*.jsonl + literature_chunk_sets/literature_chunks`。
- `docs.jsonl` 与单一 `chunks.jsonl` 已退出 PDF 主链，不再作为正式输入。

使用建议：

- 批量结构化时优先给 `PDF文件转结构化数据文件` 提供 `content_db`，让事务自动回写 `structured_abs_path`。
- 做分块时优先传 `content_db` 或 `input_structured_dir`。
- 做单篇粗读、单篇精读、综述草稿生成时，优先传 `input_structured_json`、`input_structured_dir` 或 `content_db`。

## 11.1 当前保留的两套 PDF 能力

当前版本只保留两套正式 PDF 能力：

1. 传统 PDF 结构化与分块套件
2. 阿里百炼多模态 PDF 解析套件

选择建议：

- 如果你的目标是进入 AOK 事务主链、写回 `content.db`、复用既有 affairs，请优先使用传统套件。
- 如果你的目标是对单篇或一批 PDF 做高精度版面理解，并拿到结构树、线性索引、重建文档和多模态 chunks，请优先使用阿里百炼套件。

### 11.1.1 传统 PDF 结构化与分块套件

适合场景：

- 批量把 PDF 转成统一 `aok.pdf_structured.v3`
- 将解析状态回写到 `database/content/content.db`
- 为后续 `解析与分块`、`单篇粗读`、`单篇精读` 等事务提供标准输入

典型入口：

- affair：`PDF文件转结构化数据文件`
- affair：`解析与分块`
- tools：`build_structured_data_payload(...)`
- tools：`build_chunk_entries_from_structured_data(...)`
- tools：`write_chunk_shards(...)`

### 11.1.2 阿里百炼多模态 PDF 解析套件

适合场景：

- 希望直接用阿里百炼视觉大模型做页级理解
- 需要输出 `structured_tree.json`、`linear_index.json`、`chunk_manifest.json`、`chunks.jsonl`
- 需要每个 PDF 一套独立输出目录，且单篇解析不混入多文献上下文

公开入口：

- `from autodokit.tools import parse_pdf_with_aliyun_multimodal`
- `from autodokit.tools import batch_manage_pdf_with_aliyun_multimodal`

后处理约定：

- 如果你要对阿里百炼解析产物再做一轮正文清洗、页眉页脚剥离或污染块剔除，请使用 `from autodokit.tools.llm_clients import postprocess_aliyun_multimodal_parse_outputs`。
- 这一步已经被 A060、A100 以及其它直接生成阿里百炼解析资产的事务自动接入，普通使用者通常不需要手工调用。
- 不建议直接导入底层 `autodokit.tools.ocr.aliyun_multimodal.aliyun_multimodal_postprocess_tools`。

最小单篇示例：

```python
from pathlib import Path
from autodokit.tools import parse_pdf_with_aliyun_multimodal

result = parse_pdf_with_aliyun_multimodal(
    pdf_path=Path("D:/papers/demo.pdf"),
    output_root=Path("D:/outputs/pdf_parse"),
    api_key_file=Path("D:/secrets/bailian-api-key.txt"),
    source_metadata={"title": "Demo Paper", "year": "2026"},
)
print(result["output_dir"])
```

最小批量示例：

```python
from pathlib import Path
from autodokit.tools import batch_manage_pdf_with_aliyun_multimodal

summary = batch_manage_pdf_with_aliyun_multimodal(
    output_root=Path("D:/outputs/pdf_batch"),
    api_key_file=Path("D:/secrets/bailian-api-key.txt"),
    jobs=[
        {
            "pdf_path": Path("D:/papers/a.pdf"),
            "output_name": "paper_a",
            "language": "en",
            "source_metadata": {"title": "Paper A", "year": "2024"},
        },
        {
            "pdf_path": Path("D:/papers/b.pdf"),
            "language": "zh",
            "source_metadata": {"title": "论文B", "year": "2023"},
        },
    ],
    output_name_key="output_name",
)
print(summary["run_summary_path"])
```

注意事项：

- API Key 只通过 `api_key_file` 路径传入，不支持把 key 文本直接硬编码进脚本。
- 单篇工具默认自动生成独立输出目录；若显式传入 `output_name`，则使用该目录名。
- 批量工具只管理 job 循环、目录命名、汇总和报告，不负责工具链路由。
